# Cough-Audio Classification Interface Prototype

> **Educational prototype only. This repository is not a medical device and must not be used for diagnosis, screening, treatment, or clinical decision-making.**

This repository demonstrates the shape of a browser-to-model workflow for cough-audio uploads. It contains front-end interface files, a Flask upload endpoint, MFCC feature extraction, and a small Random Forest training scaffold built with synthetic data.

All training features and labels in the checked-in training script are randomly generated. Therefore, the model has no medical validity and any displayed “Healthy” or “Ill” result is only a mocked interface output.

## Components

### Frontend

The [`frontend`](frontend) directory contains interface files for selecting an audio file, submitting it for analysis, showing loading feedback, and displaying the returned label.

### Flask API

[`backend/app.py`](backend/app.py) defines a `POST /predict` endpoint. It:

1. checks for a multipart field named `file`;
2. sanitizes and saves the uploaded filename;
3. calls `predict_cough` on the saved file;
4. returns a JSON `prediction` field.

### Feature extraction and model scaffold

[`model/preprocess.py`](model/preprocess.py) loads audio at 22,050 Hz, computes 13 MFCC coefficients with `librosa`, and averages each coefficient over time.

[`model/training.py`](model/training.py) creates 100 random 13-dimensional feature vectors and random binary labels, applies an 80/20 split, trains a scikit-learn `RandomForestClassifier`, and serializes it as `cough_model.pkl`.

[`model/predict.py`](model/predict.py) loads that serialized model, extracts MFCC features, and maps class `0` to `Healthy` and class `1` to `Ill`.

## Repository structure

```text
frontend/   upload and result-display interface
backend/    Flask upload endpoint
model/      MFCC extraction, synthetic training scaffold, and inference helper
```

## Current reproducibility gaps

This repository is not currently packaged as an end-to-end runnable application:

- no dependency manifest or environment specification is checked in;
- the serialized `cough_model.pkl` artifact is not present in the repository and must be generated by the training scaffold;
- module paths in the Flask backend and model directory need to be reconciled for one consistent package layout;
- no integrated start command or test suite is documented.

## Safety, privacy, and model limitations

- The training data is random synthetic data, not cough recordings or clinical labels.
- Accuracy on randomly generated labels is not a meaningful model-quality or health metric.
- There is no clinical validation, external test set, calibration, sensitivity/specificity analysis, demographic analysis, or robustness study.
- Uploaded files are written to disk by the current Flask demo; the code does not define retention, deletion, encryption, authentication, file-size limits, or content validation.
- The Flask app runs with debug mode enabled in the checked-in entry point and is not configured for deployment.
- The binary `Healthy`/`Ill` labels do not correspond to a defined medical diagnosis.
- The repository name and earlier project framing do not establish organizational endorsement or deployment.

Before this could become a meaningful ML demonstration, it would need a permitted and documented dataset, a defensible prediction target, leakage-safe evaluation, calibrated metrics, robust privacy controls, and review by appropriate clinical and security experts.
